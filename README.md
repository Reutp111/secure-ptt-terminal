\# Secure PTT Terminal\*\*Secure PTT Terminal\*\* is a terminal-based proof-of-concept communication system for temporary, channel-based text communication with local Text-to-Speech playback.The project demonstrates a lightweight and privacy-aware communication model where multiple authorized clients can connect to a central relay server, exchange short text messages in real time, and have incoming messages read aloud locally on the receiving machine.The system is designed as a legitimate secure communication prototype, not as a remote shell, backdoor, or command execution tool.\---\## OverviewSecure PTT Terminal provides a simple terminal interface that behaves similarly to a lightweight Push-to-Talk / Walkie-Talkie system, but instead of transmitting microphone audio in the current version, it transmits text messages that are converted into speech on the receiving client.The main idea is:```textUser writes a short message&#x20;       ↓Message is sent to a relay server&#x20;       ↓Relay forwards it to clients in the same channel&#x20;       ↓Receiving client displays the message&#x20;       ↓Receiving client reads the message aloud using local TTS```The relay server does not store message content in a database or file. It only forwards messages to currently connected clients in the same channel.\---\## What Makes This Project UniqueThis project combines several ideas into one simple and practical prototype:\* Terminal-based communication interface\* Real-time WebSocket messaging\* Channel-based user separation\* Local Text-to-Speech playback\* Temporary message display\* No database requirement\* No message content storage on the server\* Cross-platform concept: Windows, Linux, Kali, and virtual machines\* Simple deployment model using a single relay server and multiple terminal clients\* Security-aware design suitable for learning, demonstrations, and further developmentUnlike a normal chat application, this project focuses on temporary operational communication. The messages are not meant to become a permanent chat history. They are meant to be delivered, displayed briefly, optionally spoken aloud, and then disappear from the visible interface.\---\## Important ClarificationThis project is \*\*not\*\*:\* A remote shell\* A reverse shell\* A command-and-control tool\* A stealth communication tool\* A tool for bypassing firewalls, EDR, SIEM, or organizational monitoring\* A tool for executing commands on remote machinesThis project is:\* A legitimate communication prototype\* A terminal-based messaging client\* A temporary text-to-talk system\* A privacy-aware proof of concept\* A foundation for secure Push-to-Talk style communicationThe system does not execute operating system commands received from other users. Any text entered inside the client is treated as a communication message only.\---\## Architecture```text┌──────────────────────┐│  Terminal Client A   ││  Windows / Linux VM  │└──────────┬───────────┘&#x20;          │&#x20;          │ WebSocket&#x20;          │&#x20;          ▼┌──────────────────────┐│    Relay Server      ││      Node.js         │└──────────┬───────────┘&#x20;          │&#x20;          │ WebSocket&#x20;          │&#x20;          ▼┌──────────────────────┐│  Terminal Client B   ││  Windows / Kali VM   │└──────────────────────┘```The relay server is responsible only for:\* Accepting WebSocket connections\* Registering users to channels\* Forwarding messages to clients in the same channel\* Printing minimal audit logs\* Rejecting invalid or oversized messagesThe client is responsible for:\* Connecting to the relay\* Joining a channel\* Sending messages\* Displaying incoming messages\* Reading incoming messages aloud using local Text-to-Speech\* Clearing temporary messages from the interface\---\## Current Features\### Terminal User InterfaceThe client runs directly inside a terminal or CMD window. It provides a simple text-based interface showing:\* Current user\* Current channel\* Relay server URL\* TTS status\* Temporary messages\* Available commandsExample:```text╔════════════════════════════════════════════════════════════╗║              SECURE PTT TERMINAL - TEXT TO TALK           ║╠════════════════════════════════════════════════════════════╣║ User: reut              Channel: it                       ║║ Server: ws://localhost:3000                               ║║ TTS: ON                                                   ║╚════════════════════════════════════════════════════════════╝```\---\### Channel-Based CommunicationUsers communicate only with other users connected to the same channel.Example:```textUser: reut      Channel: itUser: kali      Channel: it```Both users can communicate because they are in the same channel.If another user joins a different channel:```textUser: admin     Channel: security```That user will not receive messages from the `it` channel.\---\### Text-to-SpeechIncoming messages can be read aloud on the receiving machine.The current version uses local Text-to-Speech through Python’s `pyttsx3` library.This means:\* The relay server does not generate audio\* The relay server does not store audio\* The relay server does not process speech\* The message is converted to speech only on the receiving clientThis design keeps the relay simple and reduces unnecessary exposure of message content.\---\### Temporary MessagesMessages are displayed in the terminal interface as temporary operational messages.The current client keeps only a limited number of visible messages in memory and does not create a persistent message history.The server does not store message content.\---\### Minimal Audit LoggingThe relay server prints minimal audit events such as:```text\[AUDIT] user\_connected user=reut channel=it\[AUDIT] message\_relayed from=reut channel=it length=24\[AUDIT] user\_disconnected user=reut channel=it```The server does not print or store the actual message text.This allows basic operational visibility without storing sensitive message content.\---\## Privacy-Oriented DesignThe project follows a “no message content retention” approach.The relay server does not store:\* Message text\* Audio recordings\* Speech output\* Chat history\* Transcripts\* Attachments\* FilesThe server may display minimal metadata for debugging or audit purposes, such as:\* User connected\* User disconnected\* Channel name\* Message length\* Delivery eventThis is not the same as “leaving no traces.”The goal is not stealth.The goal is to avoid storing message content while still keeping the system legitimate and auditable.\---\## Project Structure```textsecure-ptt-terminal/│├── server/│   ├── server.js│   ├── package.json│   └── using it in a real organizational environment, the following security controls should be added:\### AuthenticationUsers should authenticate before joining a channel.Possible options:\* Username and password\* JWT tokens\* API keys\* SSO integration\* Device-based authentication\### TLS / WSSThe current local version may use:```textws://```For real use, it should use secure WebSocket:```textwss://```This requires TLS certificates and secure server deployment.\### End-to-End EncryptionIn the current proof of concept, the relay forwards message content.For stronger privacy, encryption should be implemented on the client side so that:\* The sender encrypts the message\* The relay only forwards encrypted data\* The receiver decrypts locally\### Role-Based Access ControlUsers should only access channels they are allowed to join.Example:```textIT users       → it-supportSecurity users → securityOperations     → operations```\### Device RegistrationOnly approved machines should be allowed to connect.Possible controls:\* Device ID\* Client certificate\* mTLS\* Signed client builds\* Allowlisted devices\### Rate LimitingThe relay should limit message frequency to prevent abuse or flooding.\### Input ValidationThe relay and client should validate message size, message type, and channel name.\### Logging PolicyThe project intentionally avoids storing message content.However, real deployments should define a clear logging policy for:\* User connection events\* Authentication events\* Channel join events\* Message delivery metadata\* Error events\---\## LimitationsThe current version is an MVP and has several limitations:\* No user authentication yet\* No TLS/WSS by default\* No end-to-end encryption yet\* No persistent user management\* No role-based channel permissions\* No production deployment configuration\* Text-to-Speech quality depends on the operating system\* No real microphone Push-to-Talk audio streaming yet\* Messages are temporary in the interface but may still exist briefly in process memory during delivery\---\## Future ImprovementsPossible future development stages:\### 1.
